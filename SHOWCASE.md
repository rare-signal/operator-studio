# Operator Studio — static showcase build

A self-contained static export of Operator Studio populated with the
agentic chats that built it. No Node runtime, no database, no API
routes — every page is pre-rendered HTML, deployable to any static
host.

## What the showcase contains

Generated by `pnpm showcase:snapshot`, baked at build time:

- **Threads** — every Claude Code transcript discovered under
  `~/.claude/projects/-Users-<you>-operator-studio/` (and the
  Claude Desktop sidecar tree), filtered to non-empty sessions.
  Currently ~307 threads / ~22k messages.
- **Work sessions** — 3-hour-idle-gap segmentation of the message
  stream, mirroring `lib/operator-studio/sessions.ts`.
- **Plan** — a fabricated plan that sketches the build trajectory in
  8 steps. The step copy is hand-written in
  `scripts/showcase-snapshot.ts`; tweak it there.
- **Workspace / counts / previews** — derived from the threads.

Everything ships as JSON in `public/showcase-data/` and is read by
`lib/operator-studio/showcase-loader.ts` at build time.

## Routes that ship

Curated subset of the live app, picked because they're the core
"watch and read" surfaces:

- `/operator-studio` → redirects to first thread
- `/operator-studio/memory` — threads dashboard
- `/operator-studio/threads/[threadId]` — chat detail (one HTML file
  per thread, ~307 of them)
- `/operator-studio/plan` — fabricated plan, no pulse data

Every other route (`activity`, `metrics`, `admin`, `inbox`, `today`,
`brief`, `sessions`, `foundry`, `search`, `plans`, `pulse`, `docs`,
plus the entire `app/(studio)/`, `app/2/`, `app/v3/`, `app/v4/`,
`app/beta-remote/`, `app/api/` trees) is excluded from the build by
`scripts/showcase-build.ts` renaming their `page.tsx` / `route.ts`
files to `.showcase-disabled` for the duration of the build, then
restoring them.

## Building

```bash
pnpm showcase:build
```

This runs:
1. `scripts/showcase-snapshot.ts` — snapshots transcripts to
   `public/showcase-data/`
2. Disables every route file outside the curated set
3. Patches `dynamic = "force-dynamic"` → `"force-static"` in the four
   shipping pages (Next.js requires the export to be a static literal
   and `force-dynamic` conflicts with `output: "export"`)
4. Runs `next build` with `SHOWCASE=1 NEXT_PUBLIC_SHOWCASE=1
   NEXT_DIST_DIR=.next-showcase`
5. Restores all disabled / patched files (in a try/finally — failed
   builds never leave the working tree broken)
6. Copies shippable artifacts from `.next-showcase/` to `out/`

Output lands in `out/`. ~206MB total, dominated by the JSON
snapshots in `out/showcase-data/`.

## Local preview

```bash
pnpm dlx serve out -p 4242
# open http://localhost:4242/operator-studio/
```

Or use any static server.

## Deploy

The `out/` directory is plain HTML / CSS / JS / JSON. Drop it on:

- **Vercel**: `cd out && npx vercel deploy --prod`
- **Netlify**: drag-drop `out/` into the dashboard, or
  `npx netlify deploy --dir=out --prod`
- **Cloudflare Pages**: connect the repo, set build command to
  `pnpm showcase:build`, output directory to `out`
- **GitHub Pages**: copy `out/` to `gh-pages` branch root
- **S3 / Spaces / R2**: `aws s3 sync out s3://<bucket>` (or
  equivalent), point CloudFront at the bucket

No environment variables required at deploy time — every secret is
already nothing because there are no secrets in a static export.

## Redaction

Two-tier scrub controlled by [`scripts/showcase-redactions.ts`](scripts/showcase-redactions.ts):

1. **Drop the whole thread** — `DROP_THREAD_IF_MATCHES` is a list of
   string / regex patterns. If any pattern matches anywhere in a
   thread's title or message bodies, the thread is excluded from the
   snapshot entirely. Use this for codenames, project names, internal
   references — anything where the thread's *subject* is the
   sensitive thing and a string-replace would leave traces in
   structure or metadata.
2. **Find-and-replace** — `REDACTIONS` is an ordered list of
   `{pattern, replacement}` pairs applied as a deep walk over every
   string field of every snapshot file *just before* it's written
   to `public/showcase-data/`. Use this for paths, hostnames, emails,
   API keys.

**Defaults shipped:**
- Drop: empty list (add your own codenames / project names).
- Replace: `/Users/<host>/` → `/Users/dev/`, host username → `dev`,
  any email → `[email]`, common API key shapes (Anthropic, OpenAI,
  GitHub, Slack, AWS, JWTs) → labelled `[redacted]` tokens, Postgres
  / MongoDB connection strings → `[redacted]`.

**To add your own:** edit `scripts/showcase-redactions.ts`, drop new
entries into either list, re-run `pnpm showcase:build`. There's a
commented "Add your own below" section in each list.

**Safety:**
- Redactions only run on the in-memory snapshot before it's written
  to `public/showcase-data/`. The source transcripts under
  `~/.claude/projects/` are never touched.
- The Operator Studio database (live Postgres or `better-sqlite3`)
  is never read or written by the showcase pipeline.
- Redactions are idempotent — running the build twice produces the
  same output.

**To verify before publishing:** the build prints a summary of
dropped threads (`X sessions after thread-drop filter (dropped Y)`)
and you can spot-check the export with:

```bash
grep -ric "<sensitive-term>" out/showcase-data/
grep -ric "<sensitive-term>" out/operator-studio/threads/
```

A clean run prints nothing.

## How the showcase build is wired

The live app stays the live app. Three small additions enable the
static branch:

1. **`process.env.SHOWCASE === "1"`** — checked at the top of each
   shipping page (`layout.tsx`, `page.tsx`,
   `threads/[threadId]/page.tsx`, etc.) to swap data sources from the
   live Drizzle queries to `lib/operator-studio/showcase-loader.ts`.
2. **`process.env.NEXT_PUBLIC_SHOWCASE === "1"`** — inlined at build
   time and read by the client shell to skip its four
   `/api/operator-studio/*` fetches and synthesize a "Visitor"
   identity.
3. **`scripts/showcase-build.ts`** — the only place that knows about
   the file-shuffle and dynamic-export-string-patch dance. Live build
   (`pnpm build`) doesn't touch any of this.

## Known gaps in v1

These are working but ugly — fix passes welcome:

- **Write-UI buttons** are still visible (Reopen, Promote, Fork,
  etc.) on thread detail. Clicking them in showcase mode no-ops or
  errors silently. Need a `READ_ONLY` context to hide them.
- **Plan view** has empty pulse panel and no fulfillments. The plan
  steps render but the "in-motion" / "covered" status badges have no
  evidence behind them.
- **Recent-chats rail** (left edge popover) likely 404s on its
  preview fetches in showcase mode. Need to pre-bake the previews
  into a static JSON the rail can fetch from `/showcase-data/`.
- **Sidebar links** to disabled routes (Activity Log, Metrics,
  Admin, etc.) lead to 404s. Either hide them when SHOWCASE_MODE or
  ship a "this view isn't included in the showcase" stub.
- **Search palette** (⌘K) likely errors when invoked.

None of these block the build or the core "browse the chats"
experience. They're polish for v1.1.
